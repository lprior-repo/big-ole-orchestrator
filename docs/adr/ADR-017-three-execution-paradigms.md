# ADR-017: Three Execution Paradigms (FSM, DAG, Procedural)

## Status

Accepted

## Context

Workflow orchestration problems do not all fit the same mental model. A payment saga is naturally an FSM. A data pipeline is naturally a DAG. A complex approval flow with conditionals and loops is naturally procedural code. Forcing all problems into one model creates impedance mismatch.

All three models must share the same event log (ADR-013), the same replay mechanics (ADR-016), and the same durability guarantees (ADR-015). The choice of paradigm affects only how the developer authors the workflow and how the engine interprets the events â€” not the underlying storage or recovery mechanism.

## Decision

wtf-engine supports three first-class execution paradigms. Each is generated by the Dioxus compiler (ADR-018) and executed by a specialized actor variant that implements the two-phase replay model (ADR-016).

---

### Paradigm A: Finite State Machine (FSM)

**Best for:** Payment flows, order processing, user onboarding, anything with explicit named states and named transitions.

**Authoring:** Visual state diagram in Dioxus designer. States are nodes, transitions are edges, effects are edge annotations.

**Event log pattern:** `TransitionApplied { from, event, to, effects }` events record every state change with the full effect list embedded.

**Replay:** The actor replays `TransitionApplied` events in order. The current state is the `to` field of the last event. Effects from old events are **not re-executed** (they are recorded in the event; replay skips execution). The actor enters Live Phase already knowing its current state with no re-computation.

**Idempotency:** A duplicate webhook triggering the same transition returns "no state change required" â€” the deterministic transition function sees the current state and the incoming event and returns the same `to` state it already recorded. No duplicate effects.

**In-memory state:**
```rust
struct FsmActorState {
    current_state: String,
    applied_seq: HashSet<u64>,
    in_flight_activities: HashMap<String, ActivityDeclaration>,
}
```

---

### Paradigm B: Directed Acyclic Graph (DAG)

**Best for:** Data pipelines, parallel fan-out/fan-in, ETL, CI/CD pipelines.

**Authoring:** Node graph in Dioxus designer. Nodes are activities, edges are dependencies. Fan-out and fan-in are explicit.

**Event log pattern:** `ActivityCompleted { activity_id, result }` events record node completion. `ActivityDispatched` records dispatch. The DAG structure itself is stored in the workflow definition (NATS KV `wtf-definitions`), not the event log.

**Replay:** The actor replays all `ActivityCompleted` events to rebuild the `completed: HashSet<NodeId>`. After replay, runs the readiness check: any node whose all predecessors are in `completed` and which is not itself in `completed` is dispatched. Parallelism is preserved across crashes: after replay, all ready nodes are dispatched simultaneously in Live Phase.

**In-memory state:**
```rust
struct DagActorState {
    dag: WorkflowDag,              // loaded from definition on startup
    completed: HashSet<NodeId>,
    in_flight: HashSet<NodeId>,
    applied_seq: HashSet<u64>,
}
```

---

### Paradigm C: Procedural Workflow

**Best for:** Complex conditional logic, human-in-the-loop steps, loops, workflows that read like code.

**Authoring:** Dioxus designer generates a Rust function body with `ctx.*` calls. The developer implements activity bodies in a separate file.

**Event log pattern:** Each `ctx.activity(name, input)` call produces an `ActivityDispatched` and (eventually) `ActivityCompleted` event. The context object is the only interface between the workflow function and the engine.

**Replay:** The actor loads all `ActivityCompleted` events into a checkpoint map (`operation_id â†’ result`). The workflow function is re-executed from the top. Every `ctx.activity(...)` call checks the checkpoint map using a deterministic operation ID (derived from `instance_id + monotonic counter`). If the result is in the map, it is returned immediately without I/O. If not, the activity is dispatched and the function suspends. The function fast-forwards to the first unresolved operation.

**Determinism requirement:** The workflow function must call `ctx.*` methods in the same order on every execution. The operation ID counter increments on each `ctx.*` call. A different call order produces different operation IDs for the same logical operations and breaks replay. The linter (ADR-020) detects violations.

**In-memory state:**
```rust
struct ProceduralActorState {
    checkpoint_map: HashMap<OperationId, Bytes>, // operation_id â†’ serialized result
    op_counter: u32,
    applied_seq: HashSet<u64>,
}
```

## Consequences

### Positive

- Developers choose the model that fits their problem domain
- All three paradigms share identical durability guarantees
- The Dioxus designer supports all three on the same canvas
- Adding a fourth paradigm does not require changing the event log or actor infrastructure

### Negative

- Three actor variants to maintain (FsmActor, DagActor, ProceduralActor)
- Each paradigm has its own replay semantics â€” more to understand
- Mixing paradigms in a single workflow is not supported (choose one per workflow type)
