# ADR-016: Deterministic Replay Model (Replay Phase + Live Phase)

## Status

Accepted

## Context

The write-ahead guarantee (ADR-015) ensures no event is lost. But durability alone is not enough — recovery must also be **deterministic**. Given the same event log, every actor instance on every engine node must arrive at exactly the same state. Non-deterministic recovery (e.g., depending on wall clock time, random numbers, or external reads during replay) would produce divergent states and break the guarantees.

This ADR defines the two-phase execution model that every WorkflowInstance actor operates under, and the determinism constraints that make it correct.

## Decision

Every WorkflowInstance actor executes in one of two phases: **Replay Phase** or **Live Phase**.

### Phase 1: Replay Phase

Entered immediately on actor startup (crash recovery or initial start).

```
1. Load snapshot from sled (if available, see ADR-019)
   → sets in-memory state to snapshot state
   → sets replay cursor to snapshot.seq

2. Create an ordered JetStream consumer from replay_cursor + 1

3. For each event in order:
   a. Check: is this event's seq in applied_seq set? → skip (idempotent)
   b. Apply event to in-memory state (pure state transition — no I/O)
   c. If event is ActivityDispatched: mark activity ID as "in-flight" (do NOT re-dispatch)
   d. If event is TimerScheduled: register timer in in-memory set (do NOT re-arm)
   e. If event is SnapshotTaken: (informational, already handled via step 1)
   f. Add seq to applied_seq set

4. When consumer reaches stream tail → enter Live Phase
```

During Replay Phase, the actor **does not**:
- Publish anything to JetStream
- Dispatch to work queues
- Call external HTTP endpoints
- Read wall clock time
- Generate random values

Side effects are recorded in the event log as what *was decided*. Replay re-derives the decision from the event log. It does not re-execute the decision.

### Phase 2: Live Phase

Entered when the event log is fully replayed and the actor is current.

```
1. Check in-flight activities: for each ActivityDispatched event that has no
   corresponding ActivityCompleted/ActivityFailed: re-dispatch to work queue
   (the worker may have completed the activity while we were crashed; the
   worker's result will arrive via the normal completion path)

2. Check in-memory timers: for each TimerScheduled with no TimerFired:
   verify the timer is still registered in NATS KV wtf-timers;
   if missing (TTL expired during crash), re-register

3. Process new incoming messages normally:
   - Apply write-ahead sequence (ADR-015) for all transitions
   - Execute side effects after JetStream ACK
   - Update NATS KV materialized view
```

### Determinism Constraints

Workflow-level code (generated by Dioxus, ADR-018) must be deterministic between context calls. Specifically:

| Forbidden | Required replacement |
|-----------|----------------------|
| `std::time::Instant::now()` | `ctx.now()` (returns logged timestamp) |
| `rand::random()` | `ctx.random_u64(seed)` (returns logged value) |
| Direct HTTP calls | `ctx.activity("name", input)` (result is logged) |
| `tokio::spawn` inside workflow fn | Side effects only via context |
| `ctx.activity(...)` in non-deterministic order | Must be called in same order every replay |

The Dioxus compiler (ADR-018) generates code that uses the context API exclusively. Hand-written procedural workflows are checked by the static linter (ADR-020).

### Replay Correctness Invariant

Given:
- The same initial snapshot (or empty state if no snapshot)
- The same ordered sequence of `WorkflowEvent` values from JetStream

The replay algorithm produces:
- The same in-memory actor state
- The same set of in-flight activities
- The same set of pending timers

This invariant holds regardless of engine version, node, or wall clock time, as long as the event log is identical.

## Consequences

### Positive

- Crash recovery is deterministic and correct on any engine node
- Time-travel debugging is possible: replay to seq N on any node
- No reconciler needed: the replay mechanism is complete recovery
- State divergence between nodes is impossible given identical logs

### Negative

- Workflow code authors must respect determinism constraints
- Generated code must be re-entered from the top on every recovery (mitigated by checkpoints in procedural model)
- Very long event logs make full replay slow (mitigated by snapshots, ADR-019)

### Relationship to ADR-017

The three execution paradigms (FSM, DAG, Procedural) each implement replay differently at the application layer, but all share this two-phase model at the engine layer.
